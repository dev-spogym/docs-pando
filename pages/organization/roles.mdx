# 역할 정의 (7종)

import { Callout } from 'nextra/components'

판도 CRM은 본사 3종, 지점 4종 총 7개 역할로 구성됩니다. 각 역할은 피트니스 센터의 실제 직무 분장을 반영하여 설계되었으며, 권한 레벨 숫자가 높을수록 더 넓은 범위의 데이터와 기능에 접근할 수 있습니다. 모든 사용자는 반드시 하나의 역할에 속하며, 역할 변경은 본사 관리자만 수행할 수 있습니다.

## 본사 역할 (3종)

본사 역할을 가진 사용자는 소속 테넌트 내 모든 지점의 데이터를 조회하고 관리할 수 있습니다. 회장은 시스템 전체 설정과 구독 관리까지 포함한 최고 권한을 가지며, 대표는 전사 운영 총괄과 함께 특정 지점 겸임이 가능합니다. 본사 직원은 결재 검토와 리포트 관리 등 본사 운영 업무를 담당합니다.

| 코드 | 한글명 | 권한 레벨 | 소속 | 설명 |
|---|---|:---:|---|---|
| `chairman` | 회장 | 100 | 본사 | 최고 의사결정권자. 전 지점 모든 권한 |
| `ceo` | 대표 | 90 | 본사 + 지점 겸임 | 본사 총괄 + 겸임 지점 직접 관리 |
| `hq_staff` | 본사 직원 | 80 | 본사 | 본사 운영 업무. 결재 검토, 리포트 관리 |

## 지점 역할 (4종)

지점 역할을 가진 사용자는 소속 지점의 데이터만 접근할 수 있습니다. 지점장은 지점 내 전체 권한을 가지며 본사 결재를 요청하는 주체입니다. 스태프는 프론트 데스크 업무(회원 접수, 결제, 출석 처리)를 담당하고, FC와 트레이너는 각각 영업과 수업이라는 서로 다른 전문 영역을 맡습니다.

| 코드 | 한글명 | 권한 레벨 | 소속 | 설명 |
|---|---|:---:|---|---|
| `branch_mgr` | 지점장 | 70 | 지점 | 지점 최고 책임자. 본사 결재 요청 |
| `staff` | 스태프 | 40 | 지점 | 프론트 데스크. 회원 접수, POS, 출석, 시설 |
| `fc` | FC | 30 | 지점 | 피트니스 컨설턴트. 상담, 영업, PT 판매 |
| `trainer` | 트레이너 | 20 | 지점 | 수업 진행, 운동 지도, 체성분 측정 |

## FC vs 트레이너

FC와 트레이너는 모두 지점에서 회원을 직접 응대하는 역할이지만, 담당하는 업무 영역이 명확히 다릅니다.

**FC(피트니스 컨설턴트)**는 신규 방문 고객을 상담하고 이용권이나 PT 패키지를 판매하는 영업 전문 역할입니다. 예를 들어, FC는 체험 방문객과 상담을 진행한 후 3개월 이용권 계약을 체결하고, 기존 회원의 재등록을 유도하며, POS에서 직접 결제를 처리합니다. FC의 성과는 본인 판매 실적으로 측정되기 때문에 본인이 상담·계약한 회원의 정보에만 접근합니다.

**트레이너**는 이미 등록된 회원에게 PT 수업을 진행하고 운동 프로그램을 작성하는 전문 지도 역할입니다. 트레이너는 본인이 담당하는 수업에 배정된 회원의 운동 기록과 체성분 데이터를 관리하지만, 매출이나 상품 정보에는 접근하지 않습니다.

| 구분 | FC (피트니스 컨설턴트) | 트레이너 |
|---|---|---|
| **핵심 업무** | 회원 상담, 영업, PT/이용권 판매 | PT/GX 수업 진행, 운동 지도 |
| **매출 관여** | 직접 판매 (매출 귀속) | 서비스 제공 (수업 인센티브) |
| **회원 접점** | 상담/계약/재등록 유도 | 수업/운동 프로그램/체성분 |
| **매출 데이터** | 본인 판매 실적 조회 | 본인 수업 실적만 조회 |
| **상품 관련** | 조회 + POS 판매 가능 | 조회만 |

## 권한 유형 (5종)

권한은 단순히 "접근 가능/불가능"이 아니라 5가지 유형으로 세분화됩니다. 이를 통해 "조회는 가능하지만 수정은 불가"처럼 세밀한 권한 제어가 가능합니다. 예를 들어 스태프는 회원 등록·수정은 할 수 있지만 삭제(탈퇴 처리)는 지점장 이상만 가능합니다.

| 유형 | 코드 | 설명 |
|---|---|---|
| 접근 | `access` | 메뉴/페이지 진입 가능 여부 |
| 조회 | `read` | 데이터 열람 가능 여부 |
| 등록 | `create` | 데이터 생성 가능 여부 |
| 수정 | `update` | 데이터 변경 가능 여부 |
| 삭제 | `delete` | 데이터 삭제 가능 여부 |

## 권한 충돌 검증

<Callout type="warning">
권한 설정 시 아래 규칙을 반드시 준수해야 합니다. 상위 권한이 있으면 하위 권한이 자동으로 필요합니다.
</Callout>

- `delete` 있으면 → `update` 필수
- `create/update/delete` 있으면 → `access` + `read` 필수

## 권한 관리 화면 사용법

### 권한 설정

설정 메뉴 → 권한 관리로 이동하면 역할별 메뉴 접근 권한과 CRUD 권한을 토글로 설정할 수 있습니다. 변경 후 저장하면 시스템이 충돌 규칙을 자동으로 검증하며, 충돌이 감지되면 저장이 차단되고 어떤 항목이 문제인지 안내됩니다.

### 직원 역할 변경

직원 상세 화면에서 "역할 변경" 버튼을 누르면 역할 선택 드롭다운이 표시됩니다. 역할 변경은 본사 역할(회장, 대표, 본사 직원)만 수행할 수 있으며, 변경 즉시 해당 직원의 접근 권한이 새 역할 기준으로 재적용됩니다.
